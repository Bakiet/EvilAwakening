Introduction

Thanks for getting this kit! We can't tell you how glad we are to finally release it to the Asset Store. It's been a long journey with a lot of hard work and also a lot of fun. Now we're extremely excited to see what you are going to create with it and to hear your feedback.

This is an early version of the manual and we have a lot more info coming up. In the meantime you may email us at support@visionpunk.com with any questions. Also, be sure to check out the extensive Troubleshooting chapter at the end of this manual.

If you have found a reproducible bug, please email your detailed reproduction case to mpbugs@visionpunk.com.

What is this?

  • An add-on that will add multiplayer to your UFPS game by hooking it up with the Photon Cloud. It aims to be a bridge connecting UFPS and PUN, and a solid foundation to keep building a multiplayer game on. At the very least, it should save you months of development time when building a multiplayer game on top of UFPS.
  • A framework of wrapper scripts allowing you to get started on a game without worrying about compatibility between UFPS and Photon Unity Networking, with lots of gameplay fundamentals already taken care of. Use it for learning, for cloud-based multiplayer games, or just as a bag of tricks for implementing your own multiplayer solution.

  • A classic Team Deathmatch demo game, extended from the core scripts with easy-to-follow code comments and instructions, featuring an awesome deathmatch level, Helipad Escape, built by the ProBuilder team

What is it not?

  • A multi million dollar game developed by a 100+ person team. This is but a humble scripting starter kit with a gameplay demo comparable roughly to Quake 1 or 2, gameplay-wise. Yeah, as in 1996 :)

  • It is by no means a zero-scripting-required in-editor tool allowing for easy drag-drop design of multiplayer gameplay. If you have game designs beyond simple Team Deathmatch - and you are not a scripter - then this is not for you.

  • Strictly authoritative or cheat resistant. It does have a lot of authoritative logic going on but it will suit casual or coop game projects more than hardcore, competitive, cyber sports ones.

Why Photon?

Because Photon Realtime is an extremely indie- and prototyping friendly service, as in "powerful, free / affordable with very little setup" ...

PUN removes much of the technical elbow grease of getting machines to talk to each other over the Internet and lets you focus on game development.

Some cool Photon facts

  • No servers needed. Runs in the Photon Cloud (EU, US & Asia).
  • Used by folks like Square Enix, 22cans and Muse Games
  • Fully integrated with Unity
  • Monthly traffic: 1,2 Petabytes
  • Largest Photon Cloud games have about 30,000 CCUs (Concurrent Users) and appr. 40,000,000 monthly active users (MAU)
  • Registered developers: 86,000 growing by +2k new sign-ups each month
  • +100 Photon Cloud servers (bare metal) in the US, Asia and Europe regions
  • #1 third party multiplayer service for Unity & in Top 20 of most popular packages in the Asset Store

We think Photon rocks, basically - we just love it and are using it in our own indie game projects.

Good luck with your game!

/VisionPunk Team

Getting started

Requirements

Be sure that you are running atleast these software versions before using the starter kit, or it won't function as intended:

  • Unity 5.2.3 or later
  • UFPS : Ultimate FPS version 1.5.2 or later
  • Photon PUN Free or Photon PUN+ version 1.56 or later is required, but you do not need to install it since a lightweight version is included in this package.
  • This asset no longer supports Unity 4. If you have downloaded a Unity 4 version of the package and require the Unity 5 version, then you may login to Asset Store from a Unity 5 editor to download it.

Installation

When running this system for the first time it is recommended to do so in a new, clean Unity project and try out the demos first.

  1. Import UFPS
  2. Import this asset, the UFPS Photon Multiplayer Starter Kit.
  3. Setup Photon PUN
  1. PUN is included in the package. This version is guaranteed to work with the starter kit. We recommend not installing other versions of PUN until you have tried things out. Just install the multiplayer kit and move on to the next step.
  2. Configure Photon with your App ID by searching the Project View for "PhotonServerSettings"
    ... or by going to the     Unity main menu -> Window -> Photon Unity Networking -> PUN Wizard -> Setup
  3. In PhotonServerSettings, make sure to set Hosting to Photon Cloud and select the Region most suitable for your location. This will affect ping times to the Photon Cloud and thereby ingame latency.
  1. Add all the included scenes to your editor File -> Build Settings. All scenes have names beginning with "UFPSMP_". For now, make sure to drag the scene "UFPSMP_TestMap" to the top of the level list (index 0).

Quickstart

This package comes with several demo scenes covering all the features. To see things in action you must build these scenes and run them on separate clients using your own app in the Photon Cloud.

Before you begin

  • See the Installation section to get your app set up if you have not done so already.
  • Make sure all the scripts and assets have been imported and compiled.
  • And ofcourse ... make sure you have a live Internet connection.

Building the Demo Game

The demo level is called Helipad Escape and it features a band of gentleman robbers duking it out against a squad of super manly SWATs.

  1. Make sure that all included scenes have been added to the editor File -> Build Settings. All scenes have names beginning with "UFPSMP_".
  2. Make sure that the scene "UFPSMP_HelipadEscape_Menu" is at the top of the level list (index 0). Also, load this scene in the editor.
  3. From the Unity main menu, Click "File -> Build Settings".
  4. In the "Platform" box, select "PC, Mac & Linux Standalone" (NOTE: this is the only supported build target at time of writing).
  5. Click the "Build" button.
  6. Start two instances of the standalone build (or hit Play in the editor for Player 1 and run a standalone client for Player 2).
  7. In the game's main menu, select a player name for each client and click "Log On" in both windows.

The clients should both connect to the Photon Cloud, load the level and spawn one player each, on the Red and Blue team, respectively.

Your players can run around, pick up weapons and collect score by shooting each other. You can press TAB to see the team + individual scores, and ENTER to chat. After 5 minutes the game will end and show the final score for 20 seconds before clearing the scoreboard and starting over.

TIP: In this test we have run the game on a single machine, but this is no different from playing with machines on opposite ends of the globe. It's all the same to Photon. You can zip up and send the standalone build to your friends anywhere and it will connect automatically in this way as long as there is an Internet connection.

Trying out the Game Mode Examples

These exist in a single demo scene that we use for development and prototyping. It is based on the UFPS CleanScene, and has two sets of team spawnpoints, and a bunch of basic weapon and ammo pickups. The Hierarchy view has a gameobject called "GameModes", with 4 children that can be activated to produce variations on the multiplayer gameplay. Similarly to the demo game:

  1. Make sure that all included scenes have been added to the editor File -> Build Settings. All scenes have names beginning with "UFPSMP_".
  2. Make sure that the scene "UFPSMP_TestMap_Menu" is at the top of the level list (index 0). Also, load this scene in the editor. If you have this set up right, the standalone build main menu should just have a blue background (no fancy robber).
  3. In the "Platform" box, select "PC, Mac & Linux Standalone".
  4. Click the "Build" button.
  5. Start two instances of the standalone build (or hit Play in the editor for Player 1 and start a standalone client for Player 2).
  6. In the main menu, select a player name for each client and click "Log On" in both windows.

The first time you build this demo it will have the Deathmatch

mode enabled. This is just like the demo game, but without teams.

In order to try out the remaining game modes: stop the game, go to the editor Hierarchy view, unfold the "GameModes" object and deactivate every child except the one you wish to try out. You must then rebuild the standalone client in order to be able to play against it with the editor.

List of Example GameModes

BasicMultiplayer

UFPS married to the Photon Cloud, plain and simply. You can log on, pickup items and kill each other but there are no rules and no real gameplay.

DeathMatch

You get awarded score for killing other players. The scoreboard will show Deaths, Frags (kills) and Score (the sum of Deaths + Frags) for each player. The game restarts after 5 minutes.

This is like DeathMatch, but players are randomly assigned to teams and contribute to a Team Score. Players are listed under their respective team in the scoreboard.

MultiTeamDeathMatch

There can be an unlimited number of teams! This mode has 4 teams, which happened to be as many as we could cram into the scoreboard :)

The SimpleChat

This scene is just a plain and good old chat! An example of how to connect to Photon with this framework without a game state or the need to have player prefabs spawned. For example, it could be used as one piece of a traditional lobby with a chat.

To build this scene, go through exactly the same steps as the previous two scenes, but add the "UFPSMP_SimpleChat" to your build settings instead.

Testing in Singleplayer

Typically you must run a level logging in from a menu scene. You can't run a level scene directly (it contains no player or camera and will just display a black screen). But logging into Photon over and over can be time consuming when you just need to tweak things like body animations, pickups or weapons.

To speed things up there is a small utility prefab called "SinglePlayerTest", a workflow tool for starting the scene quickly. When this gameobject is active on startup all multiplayer functionality gets disabled.

  • If the component's 'SpawnMode' is set to 'Prefab' (and a valid player prefab from the Project View is provided) it will be spawned in the scene.
  • If 'SpawnMode' is set to 'Scene', the first player object found in the scene will be used.

A SinglePlayerTest gameobject is included in each gameplay demo scene. If you want to use this functionality in your own scenes, you may drag the "SinglePlayerTest" prefab into your scene. Just remember to disable it before running the game in multiplayer. Also, note that this script is only intended for testing, and not for use in an actual game.

Framework overview

The core framework supports most of the functionality seen in the demo game, such as bullet gameplay, teams, pickups and respawning, but the team and score logic of the Deathmatch mode is actually implemented as a mod!

The deathmatch scripts can be used as a base for your own more advanced game modes, but you can also disregard them completely and take the system in entirely different directions. Perhaps you mean to build an online 3D store, a Virtual Reality architectural visualization or a Second Life-like? Just delete / disregard any scripts with the prefix "vp_DM" (for deathmatch) and keep the ones prefixed "vp_MP" (for plain multiplayer).

Master

This folder contains the scripts that drive basic multiplayer game logic and game state. Most importantly, "vp_MPMaster" (which is king of the jungle and the closest thing to a server) and also scripts for establishing the Photon connection, keeping track of time, pickups, damage and death allocation, player spawning and basic team logic.

Player

This folder contains scripts pertaining to players in multiplayer. These scripts deal with the local / remote player concept, colliders, player stats, player visual representations (player types), nametag rendering and more.

Demo

This folder contains scripts that were written for demo purposes only.  You can delete this whole folder and the underlying multiplayer framework will still be functional.

Demo/Deathmatch

These files inherit a number of master scripts to implement new player stats and team logic typical for Team Deathmatch. They are intended as a straightforward example of how to extend the base "vp_MP" scripts for new gameplay.

Demo/Gui

Contains a few GUI scripts used in the demo, such as a simple main menu, a chat and the scoreboard. These scripts were not written to be incredibly reusable but parts of them might be educational / be of interest.

Utility

This folder has a few simple but very useful scripts to facilitate multiplayer testing and design workflow.

Important Concepts

(this section is under construction)

Photon Cloud network model

Photon Cloud is a Software as a Service (SaaS) and as such a fully managed service. You can completely concentrate on your application client while hosting, server operations and scaling is all taken care of by Exit Games. The Photon Cloud scales automatically to accommodate all your users, using an efficient load balancing workflow. Various subscriptions allow you to handle your monthly active users count (Photon Realtime has a free plan for up to 20 CCU which is usually more than enough for prototyping).

Unlike many traditional multiplayer games (and certainly most FPS games) the Photon Cloud does not lend itself to the classic Client / Server model (see these links for more details). This may seem counter intuitive and unusual, but it also opens up new possibilities and entirely new ways of designing for multiplayer.

For example, instead of a "server", there is a "Master Client". If lightning strikes the Master Client's real-world house and he's forcibly disconnected, a "Master Client handover" occurs. This means that the next player in line becomes Master (becomes authoritative).

The kinda cool thing about this is that you don't need to setup servers running your game around the clock in order to support multiplayer. If noone in the whole wide world plays your game and one person suddenly logs in, a server will boot up at that very moment. If another person joins they will be matched, and if more people join and the first guy drops off after a few hours the game just keeps running, until the last person disconnects and the whole thing goes back to rest.

However, it comes with its own unique set of challenges and ways of designing for reasonably authoritative multiplayer.

The Master Client

In the UFPS implementation, the Master Client handles multiplayer game time cycles, assembles and broadcasts full or partial game states with game phase, clock and player stats and allocates team and initial spawnpoint to joining players. The master role is handed over to the next person when the master quits the game.

In UFPS, the game logic running on the Master Client's machine is the only thing that can kill a player. This always happens via an RPC sent out by the master.

Distributed game logic

Because any player could become master when least expected, every player needs to keep a complete copy of the simulation at all times. In the UFPS implementation, the scene and all the gameobjects and components inside of it constitues this full simulation. For example: the damagehandler components on every player holds their amount of health points, and their inventory components know what each player carries.

In reality, because of sync issues there may be slight deviations between the various scenes. When a handover occurs the new Master Client always broadcasts its version of the simulation as the new "law of the land". If everything is implemented nicely, in 99 times out of 100 any deviations should be unnoticeable.

Because every client has a copy of the entire simulation, and knows about what the others can and can't do, it is possible to implement various schemes of cheat detection and statistics to collectively detect and ban cheaters. Ofcourse, it's also limiting in some respects, for example: gameplay that relies heavily on secrecy may be quite easy to exploit by advanced cheaters.

For more information about how distributed gamelogic hampers cheating, see this chapter.

Separation between UFPS and PUN

Base UFPS does not talk directly to PUN. There are absolutely no references to Photon in the UFPS code base. The reason for this is it should be possible to adapt UFPS to any networking system if needed.

In other words: our multiplayer add-on is a bridge or layer of separation between UFPS and PUN. UFPS never references multiplayer scripts directly. Instead it sends an event that is picked up by the multiplayer add-on (or your own network bridge) and forwarded to the network.

Transmit / Receive naming convention info

"Transmit" and "Receive" are method name prefixes with the purpose of clarifying network code flow. These are used for one-shot incidents (as opposed to every-frame-Update-code) that the master needs to tell all the other clients about in an authoritative manner.

Transmit method

  • Broadcasts a change that occured in a local master scene, to all remote machines / scenes.
  • Usually a vp_GlobalEvent target method on the master.
  • Typically sent by a UFPS Base script and picked up by an UFPS Photon Kit (or custom) script which sends out a corresponding "Receive" RPC.
  • Only ever executed by the master client.

Receive method

  • Reacts to, and implements a change described by the master in a local, non-master scene.
  • Usually a PunRPC (Remote Procedure Call) method on the client.
  • Typically declared a Multiplayer Kit script (or your own custom network script).
  • Always aborts if sender is not the master, and sometimes aborts if target is the master (in cases where master already imposed the change on itself).

Example

  1. An incident takes place in the master's scene. For example: an object gets killed.
  2. A single & multiplayer-agnostic script in UFPS Base determines that the local machine is a multiplayer master, and sends a vp_GlobalEvent with the "Transmit" prefix. In this case: "TransmitKill".
  3. A master script of the Multiplayer Kit listens for the UFPS "TransmitKill" vp_GlobalEvent, and sends out an RPC to all non-master clients with the prefix "Receive", for example: "ReceiveKill".
  4. The "ReceiveKill" RPC is triggered in a client script of the Multiplayer Kit which in turn recreates the kill on the relevant UFPS Base object in the local client scene.

Game State

The "Game State" is a hashtable sent out by the master client, listing some important game phase stats like time left and standard duration, along with all (or just some of) the stats of all (or some of) the players.

The game state is used rather than Photon's Room / Player Properties, because by default any player can change those properties in Photon and we can't allow that.

  • Player stats are stored in a vp_MPPlayerStats- derived component on every player
  • The Master Client will assemble a gamestate for purposes of pushing its version of the simulation onto all other players
  • If a client suddenly becomes master, it will assemble a game state based on the vp_MPPlayerData and vp_MPMaster components in its own scene and immediately push it to all other players
  • When game states are sent out, non-master clients must accept these new stats, no questions asked.
  • A game state gets broadcast ...
  • ... when a player joins
  • ... when a player disconnects (and thereby, when a master client switch occurs)
  • ... when a player gets killed
  • The master pushes out a gamestate via BroadcastGameState, which ...
  • ... scans all network players in the master scene using AssembleGameState
  • ... sends the RPC SyncGameState to all players
  • ... the client method SyncGameState loops all network players in ... the client scene and imposes the stats received in the RPC call

Game Stats

Phase

The game state can exist in three basic phases:

NotStarted

We haven't arrived into the game yet (loading level / logging on / quitting).

Playing

A match is ongoing and we're in it!

BetweenGames

The current battle has been fought and we're waiting for the next one to begin. In Deathmatch, this is where we show a scoreboard.

TimeLeft

Time left in this game (if 0, game runs forever).

Duration

The duration of a typical game.

Player Stats

The vp_MPPlayerStats component is a hub for all the common gameplay stats of a player in multiplayer. These are regulary gathered by the Master Client and pushed onto other clients which must update their scenes accordingly.

ID

The Photon player id

Name

The Photon player name

Position

The serialized position of the player

Rotation

The serialized rotation of the player

Inventory

The list of item records that a current player is considered to possess. Regulated by the master client

Shots

This is the amount of unique fire events triggered over the network by each specific player. It is used to establish a deterministic seed for distributed game logic.

Health

The current health of the player's damagehandler. This is used by the master client to sync the damagehandler in the master scene with all other scenes, for example: when a new player joins

PlayerType

This determines the player prefab that will be spawned for a player

It can be used for having different player body models per team, or could be extended to mean different models per team and class

Team

This can be used to divide players into groups. the default team is 0, which means "NoTeam"

Note that team does not need to have to do with competitive multiplayer.

Examples:

  • in regular Deathmatch everyone is on team 0, fighting everyone
  • in Team Deathmatch there is atleast two teams fighting each other
  • in a Coop game the players could be team 1, the monsters team2 and NPCs could be team 0.

Team could also mean faction, race, occupation, clan et cetera, all depending on your game design.

Weapon

The current weapon wielded by this player

Prediction

The multiplayer kit features experimental prediction of player positions. This is used to update player positions smoothly, accurately and reliably even under heavy lag.

This works by using the last reported velocity of the player to figure out where it's likely to be positioned in the next frame. While things always behave a bit oddly under lag, movement will be nowhere near as buggy with prediction.

The easiest way of getting a feel for what prediction does is to see it in action.

Testing the prediction

  1. Build the "UFPSMP_TestMap" demo to Standalone as explained here.
  2. Start one client running in the editor, and another running the standalone build.
  3. In the Hierarchy view, click on the remote player (the player whose name does not begin with "LOCAL")
  4. In the Inspector, unfold the "vp_MPRemotePlayer" component and check the "Show Network Position" checkbox. This will reveal the actual, non smoothed network position of the remote player by means of a white, transparent capsule.
  5. Run around (hold down Shift) with the standalone build player, and watch it in the editor game view. Notice how, while the capsule of the actual network position stutters, the player remains smoothly in its overall center.

Remote player with prediction enabled

(remaining smoothly at the correct position)

  1. Uncheck "Predict Position" on the remote player, and notice how the player will now begin drifting behind the capsule. This is what happens when you simply interpolate (Lerp) the incoming network position. While movement is smooth, the position is inaccurate. This will increase problems with bullets hitting a player on one machine but missing on another machine.

Remote player using smooth interpolation only

(drifting behind the correct network position)

  1. Let's simulate some lag! With "Predict Position" still disabled, put "40" in the "Simulate Lost Frames" field and run around with the standalone player. You will notice how the remote player now starts moving erratically, sometimes falling behind, sometimes dashing up to the capsule. Even worse for reliable bullet gameplay!
  2. With "40" still in the "Simulate Lost Frames" box, re-enable "Predict Position" and notice how the player body starts moving much more reliably even if the capsule now updates much more roughly.

This prediction model is still very simple and a lot remains to be done here, but as you can see anything is better than "plain Lerp" =).

Good to know

If you are planning to add custom gameplay, then scripting skills are required!

  • You will need a decent understanding of Unity, C#, UFPS and Photon concepts. Read this manual carefully and study all the code comments throughout the scripts.

  • If you have ideas for complex multiplayer gameplay designs, then keep this in mind: They say programming is an art form that fights back. Multiplayer programming is an art form that fights back with a vengeance! Think of your game design like a mechanism: the more moving parts in your mechanism, the more often it is going to break. Keep it simple.

  • It is strongly recommended to form an understanding of how to properly test and think about multiplayer game logic. This will save you tons of time and will make the whole development process a lot more enjoyable ...

If you are planning to add custom player models and /or weapons:

  • Artists will want to learn about the UFPS Player Body system and Weapons systems.

  • Scripters should study the UFPS manual Body scripting , Shooters and Inventory chapters.

Scene Setup

Demo scenes

  • To make the included demos run in multiplayer, you must start it from the corresponding "_Menu" scene.
  • You must add all scenes to the build settings before loading will work (!).
  • The correct menu scene must always be the topmost one (index 0), in order for the menu to display in standalone builds.
  • Activate the SinglePlayerTest gameobject to directly test demo maps in singleplayer. Without it, the scenes will just render black (they have no player or camera).

Custom scenes

  • GUI and master gameplay scripts should be kept in a separate scene that is used to login, host and join games.
  • Actual levels (containing architecture, terrain, props, pickups etc.) should be kept in separate scenes with no player or camera objects and no master scripts.
  • Important: The string "CurrentLevel" on the vp_MPMaster component must be set to an existing level name on startup in order to host a game (for the master to be able to login itself).
  • Naturally, the exact same levels must exist on all clients.
  • You can drag the SinglePlayerTest prefab into your scene to quickly test it in singleplayer. Don't forget to set up its spawn parameters.
  • Menu and master gameplay objects need to have unique PhotonView IDs. This is to prevent collision with the PhotonView IDs of props in level scenes. Tip: Set them to 901, 902 etc.
  • If a scene has UFPS spawnpoints, these will be detected and used by the multiplayer kit. Spawnpoint names will automatically be mapped to multiplayer teams defined in the team manager script. Teams that can't find a corresponding spawnpoint name will fallback to other existing spawnppoints (if any). If there are no spawnpoints, players will spawn at the world origin (0, 0, 0).

Master Setup

Every startup / main menu scene should have a "Master gameobject". In the Helipad Escape menu scene, this object is called "HelipadEscapeGame". The GameModes menu scene actually has several of these (childed under the GameModes gameobject), which is perfectly recommended. What you'd typically do in a game project is keep your game's different gamemodes stored just like that and enable them according to input from some game mode choice menu.

These are the objects that will typically be seen on a Master gameobject:

Checklist

vp_MPConnection

Initiates and manages the connection to Photon Cloud

vp_MPPlayerSpawner

Handles multiplayer instantiation and destruction of player gameobjects. Important: This component must be added to the gameobject before / above the vp_MPMaster component.

vp_MPMaster

Manages overall game logic on the master client and imposes the game state on all other clients

vp_MPPickupManager

Player Setup

The UFPS player object is quite advanced, and in multiplayer it gets a bit more complex for sure. Usually, though, when everything is set up you can re-use much of your previous gameobject hierarchy.

One thing to keep in mind is that most of the heavy setup needed is regardless of multiplayer (you need it for singleplayer too). The idea is that once you have a working 1st and 3rd person player, this object will be used to derive all other settings, most of which are handled more or less automatically.

One example of this is the Generate Remote Player wizard. The idea is that the output of this wizard needs never be edited. Ideally, you should be able to delete it at any time, regenerate it and be instantly good to go (it should be "worth nothing").

To this end, the vp_MPPlayerSpawner component has 'Add Prefabs' and 'Add Components' foldouts, listing objects that will be auto-added to local vs. remote players immediately after instantiation. This is a workflow optimization feature that allows using the plain local player prefab + a plain wizard-generated remote player prefab without further adjustments, potentially saving you tons of time!

Checklist

  1. Create a 1st person player body setup and test it thoroughly in 1st and 3rd person.
  2. Create a remote player using the Generate Remote Player wizard
  3. Make your local and remote players into prefabs in the Project View
  4. Declare a vp_MPPlayerType object (go to "UFPS -> Multiplayer -> Create Player Type"). Make sure to assign your local and remote player prefabs to this player type object.
  5. Add your player type to the Available Player Types of the  vp_MPPlayerSpawner component in your scene.
  6. Define what objects your vp_MPPlayerSpawner should add to players automatically on spawn, by unfolding:
  1. Add Prefabs
  1. Remote -> Should have a NameTag prefab set to the transform named "Head"
  1. Add Components
  1. Local -> Should define the "vp_MPPlayerStats" string (default) or "vp_DMPlayerStats" (for deathmatch)
  2. Remote -> (same as Local)
  1. Create Spawnpoints tagged with the exact name of the teams you will be using

After using the remote player wizard

Be mindful that preset files are dumped for the remote player in the folder "UPFS/Base/Scripts/Presets/3rdPerson". When you are happy with the remote player you should locate the presets that are actually used on your prefab. This can be done by dragging the prefab into the scene and right clicking on each preset text file -> "Find References Iin Scene". It can also be done by unfolding the "State" foldout on your vp_Components (most importantly the remote player's shooters) and clicking on the filenames to locate the files in the project view. At some point you should move the used files to a more permanent location where they do not risk being deleted in a cleanup round. Why? If these files are accidentally deleted, important states will be empty on the remote player and you will start to see sync issues in multiplayer (bullets ending up in different places on different machines and so on).

Bullet paths & accuracy

In UFPS, whenever a shot is fired, a random spread is imposed, making the bullet stray a little bit. This is used to make weapons more or less accurate, and to make the player fire more or less accurately in its different states (running, crouching, sliding).

Hacking bullet spread would allow a player to use weapons of 100% bullseye accuracy. To prevent this, random accuracy is calculated locally on each machine, using a deterministic seed based on the amount of times each player has fired a weapon.

This is what the "Shots" player stat is for. It is iterated every time a player fires a weapon (once per weapon discharge) and is guaranteed to be the same on all machines without sending possibly hacked data over the network. This system is very reliable and will have bullets flying along exactly the same paths on all machines as long as weapon states are in sync.

If you hack the game to make your weapon more accurate - let's say you set the bullet spread of the shotgun to zero - on your machine you will fire bullseye shots every time, but on every other person's machine the spread will be normal, since the random spread is not sent over the network - it is recreated locally on each machine using the same seed.

Inventory

Unlimited ammo

If you hack the game to get unlimited ammo you will be able to fire hundreds of bullets on your own computer - but on every other person's machine your gun will just go *click*  *click  *click*, since the inventory systems on their version of your player object won't allow your gun to fire. Any other type of item will be subjected to similar limitations.

If client is hacked to have unlimited ammo
  • Behavior on local (hacked) machine: the player will appear to have unlimited ammo, but once the "real" ammo has run out, noone will take damage when fired at, even though bullets are fired locally.
  • Behavior on all other machines: hacked player will run around dry-firing after "real" ammo has runs out, and will expose themself to risks and look really silly since noone takes damage.
If master is hacked to have unlimited ammo:

Behavior on all machines: master will have truly unlimited ammo, opponents will die when fired at, even with illegal bullets. In short: master player will be able to own everyone else.

In summary, all this will likely provide decent protection against casual hackers and off-the-shelf "cheating apps". As you can see, however the system is extremely vulnerable to the concept of a hacked master client. On the other hand, this is the case with every server-client based architecture too, i.e. trust in the server is of paramount importance. If you worry about hacked masters, then one way around it is not allowing players to act as masters, but instead to host games thru only 100% trusted master clients / host your own Photon servers.

Firing rate

The UFPS multiplayer firing algorithm caches / cueues shots at their fired positions, but subjects them to firing rate remotely on all other machines. If you hack the client to fire 30 shots from a certain position in one frame, your next 30 shots will be fired from that position only on remote machines - but at the 'legal' firing rate only and over the course of the next few seconds, regardless

of your current position. If the shots are only allowed to fire with a certain distance from your body (not yet implemented) you will arguably be hard pressed to exploit this in any meaningful way.

In short - in the best of worlds the only thing a casual hacker will succed in doing is running his own client badly out of sync - rendering himself "blind" in a sense to what  is actually going on in the game.

Level Loading

Functionality

The UFPS multiplayer framework supports runtime level loading in the form of three basic rules:

First player dictates the start level

  • The initial master (defined as the first player in a room) will make everyone else in that room load its own level.
  • Important: The public string "CurrentLevel"on the vp_MPMaster component must always be set to an existing level name before pressing "Login" in order for a master to be able to login itself.

Joining players will load the currently played level

  • The public string "CurrentLevel" is kept up to date and sent from the master to all joining players so they can load the correct level.
  • If a master switch occurs, the CurrentLevel will be correctly retained on the new master as long as it is in sync.

The master can force a level switch

  • The vp_MPMaster script has a public method "TransmitLoadLevel", accepting a string level name. At any time, this method can be called by the master to make everyone load a new level.
  • After a level load, the game mode is automatically reset (match restarts).

NOTE: While these features are technically / functionally supported, they are not exposed in the demo guis. Therefore, some light custom gui scripting is needed to make them configurable at runtime.

Scripting and level loading

Some objects are retained across levels

Some objects will survive a level load by default. This is very important to keep in mind, especially when setting up scenes and scripting gameplay. The affected objects are:

Players

  • The local player object, once spawned, will always survive until explicitly destroyed by you in script.
  • Remote player objects will be destroyed automatically when leaving the game (but not on level load).

Demo menus and master gameplay objects

  • Prime examples are "DemoGUI" and "GameModes" in the "UFPSMP_TestMap_Menu" scene. Their child gameobjects and components should always exist in a scene by themselves, used to start the game and create the login.
  • Level scenes however (such as "UFPSMP_TestMap") should never include such menu and master gameplay objects. Otherwise, after loading a level you may end up with duplicate PhotonView IDs, gameplay scripts trying to do the same thing and resulting bugs and crashes.

Menu and master gameplay objects need unique PhotonView IDs

This is to prevent collision with PhotonView IDs in level scenes. For example, if the PhotonView ID on your main menu gameobject is identical to the ID of, say, a crate in a level that gets loader later, then PUN will post an error.

Players are usually OK since they automatically get PhotonView IDs assigned starting at 1001 for player one, 2001 for player two etc. Regular scene gameobjects automatically get assigned IDs starting at 1, which is usually the problem. One thing you can do to avoid this is set a very high number (below 1000) for any standard gameobjects that are to be retained between levels. For example: 900-999, minimizing the risk of colliding with things like in-level rigidbodies.

Disabling DontDestroyOnLoad

If you want to write your own custom level loading logic that relies on disconnecting and / or destroying objects between levels, you may want to alter the default behavior. There is a toggle in the Inspector on the affected scripts (vp_MPLocalPlayer, vp_MPRemotePlayer, vp_MPConnetion, vp_MPDemoMainMenu) called "DontDestroyOnLoad" (default: ON) allowing their root gameobjects to survive between level loads. Disabling this is not recommended and only provided for advanced scripting and modding purposes.

Use OnLevelWasLoaded instead of Start / Awake

If gameplay functionality of some kind breaks after loading a level, or objects become null after loading a level, verify that any gameobject or component caching (scanning the scene to store objects) happens in OnLevelWasLoaded - never in Start or Awake, especially if a script is set to "Don'tDestroyOnLoad" and if it stores objects that are in any way specific to a level. Lists and dictionaries that store data about the level should be cleared and refilled in OnLevelWasLoaded.

Making timers survive a level load

If a necessary vp_Timer stops working after loading a new level, you may need to disable vp_Timer's "CancelOnLoad" parameter, like so:

// global variable

static vp_Timer.Handle myTimer = new vp_Timer.Handle(); // create a timer handle

// inside your method

vp_Timer.In(1.0f, ()=>

{

        // do something

}, myTimer);                        // <- hook the timer handle

myTimer.CancelOnLoad = false;        // <- do this right after calling "vp_Timer.In"

Multiplayer test checklists

Multiplayer is much harder and very different to develop and test for than singleplayer. One reason for all the extra work is that every action performed by a player (which may potentially affect another player) must be tested in a lot of different cases.

Divide & conquer: Develop and test one feature at a time. Make sure each territory is "completely conquered" before moving on to the next. You don't want multiple issues causing diffuse and hard to find bugs at once.

If you don't develop a good testing routine and a clear understanding of all the different "problem spaces", some multiplayer bugs will appear downright mystical in their behavior and may become extremely hard to find. Time spent learning to understand what can typically go wrong is time very well spent and will save you tons of work.

Troubleshooting

Connection

Trouble connecting. The logon sequence keeps aborting and retrying

The framework has a feature to abort stalled connections. This timeout may need tweaking. On the vp_MPConnection component, you can increase the amount of seconds for "Log On Timeout". This makes the game abort a connection attempt that takes too long, and automatically retries connecting (this could be exposed to your game interface, allowing the user to find a value that suits their connection). In our tests, connection has rarely stalled for more than 5 seconds, but on a choppy connection this is probably way too brief and will cause the game always to retry and never to connect. If so, try with larger values (perhaps 10 to 30 seconds).

Trouble connecting. The logon progress keeps retrying at "ConnectingToNameServer"

  • Try temporarily disabling your firewall. If the problem immediately goes away then re-enable your firewall and in its settings give the Unity Editor + your standalone builds full rights.
  • If adjusting the firewall did not work, you can try having Photon use TCP protocol instead of UDP. In the Project view -> Photon Unity Networking -> Resources -> select "PhotonServerSettings" and change the protocol to TCP. This might solve the problem.

Player

My remote players float about two inches above the ground

Set the YOffset parameter of their vp_BodyAnimators to the character controller's negative skin width, i.e. -0.08.

My remote player is not animated, and it spawns off in the distance and comes sliding into the map

Verify that the remote player has a vp_BodyAnimator component and not a vp_FPBodyAnimator which is for first person

Things seem to work fine in singleplayer mode, but when I run it in multiplayer some feature of the player does not work

Verify that the local and remote player prefabs are up to date (make sure they are activated and press "Apply" on them in the Inspector). Then, if you keep them in the hierarchy, don't forget to deactivate them again before you launch multiplayer.

In the standalone build, remote players no longer appear

Make sure you didn't build the game with an active vp_SinglePlayerTest prefab in the scene.

A component derived from vp_Component just won't register with the event handler

Is the component spawned at runtime from a separate prefab? vp_Component is hierarchy aware and will look for an event handler in the root object at runtime. If your object is moved into the player hierarchy "too late", the event handler will be null.

The player sometimes takes falling damage without a pain flash, and sometimes there's a pain flash but no falling damage

This is likely due to a too low falling damage impact threshold on the player damage handler components (it should be atleast 0.25 with the default gravity settings). This issue can occur because fall impact effects like camera shake and pain flash are calculated on the client, but the actual damage will be decided by the master. Since falling impact will diverge ever so slightly between the master and the client, the problem will be more pronounced with lower falling impact thresholds, and will wear off with higher thresholds. Note that changing things like gravity and jump force will also affect these things.

Weapons and Inventory

Ammo / weapons stop working after the player has died and respawned

Do you have a custom script that caches a UnitBank? Instances of unitbanks are created at runtime and reset upon death. A respawned player has a new instance of the same type of unitbank, and the old reference may not be null but will point to a unitbank that is no longer used.

When a player switches to a certain weapon, other players see him switching to a different one

When the local player switches to a certain weapon, it switches to a different one in 3rd person

Verify that the localplayer 1st person weapon collection is exactly the same as its remoteplayer weapon collection. When adding or removing a weapon from the 1st person local player, always remember to also add or remove that weapon to/from:

  • the localplayer 3rd person weapon group
  • the remoteplayer weapongroup

Also, try to run a standalone, pickup some weapons and connect to the game with the editor. The console should then display any errors related to remote player weapons

verify that all weapon itemtypes are present in the itemgiver component

My remote player can't pick up items

Does your remote player 'BodyCollider' have a rigidbody? Otherwise triggers won't work with it since it has no charactercontroller. Make sure the body collider has a rigidbody component that is set to 'IsKinematic'. NOTE: local player prefabs should NOT have a rigidbody since they already have a CharacterController.

Explosions don't occur on the client after a little while

This can happen if inventories are out of sync. For example: for some reason a pickup respawns on the master but not on the clients. The master player picks up a grenade and tosses it. On all the other machines there will be no grenade, so those machines won't update the master player's inventory. The grenade thrown fails to spawn except for inside the master scene. When it explodes, players will be damaged in the master scene and will loose health on all machines because of the master's authority on damage, but explosion effects will fail to spawn execpt on the master. Solution: solve the pickup respawn issue.

I can't switch to a certain weapon in multiplayer

Try disabling the inventory component on both the remote and local player prefab, rebuild the game and see if you can switch to the weapon now. If you can, make sure you have created an item type, and put the item type on a vp_ItemIdentifier on the weapon, and put the item type object on the vp_MPItemList component of your game. Re-eneable the inventory on both player prefabs, rebuild the game and try again.

A player is reloading, but the reload does not take effect on the master client. Player can fire locally but dry-fires on all other machines.

Verify that vp_WeaponReloader components are present on all the weapons in the 3rd person weapon group on both the localplayer + the remoteplayer prefabs.

My weapon seems to have no spread even though i've set it

Are you running the "vp_SinglePlayerMode" script? Since it runs multiplayer but kinda doesn't, its "GetFireSeed" delegate is empty and it hasn't gotten a spread value from the server. The weapon will have correct spread if the script is disabled, and you run the game in multiplayer.

Animations are not playing on the weapon model

Make sure that the weapon prefab has an "Animation" component, and that it lists all the animation clips that the weapon is supposed to play. For reference, study the included weapon prefabs.

Projectiles

Bullet paths are out of sync / bullets end up in different places on different machines

This can happen if a weapon has more than one shooter component for some reason.

A projectile is not spawned where a player stands, but where it used to stand

This can be the case if a player is allowed to fire on a local machine, but another client's simulation does not think it should be allowed. Try to disable the inventory component on both the local and remote player prefabs and try again. If it works on all machines with no active inventory on the players, this is a strong indication that the different player inventories are out of sync. The firing in different locations happens because of the shooting logic of the remote player class. since e.g. machinegun bullets can be fired sequentially at a very high and regular rate, but RPCs do not necessarily arrive at the same rate, but rather an irregular one, we can't rely on the firing rate logic of the vp_Shooter class (nextallowedfiretime) being in sync on all machines. Therefore, remote shots are cached and allowed to arrive in tighter clusters, or more far apart in time than the scripted refire interval. As long as the inventories are in sync, every bullet is guaranteed be fired from the exact same spot and in the same direction on all machines, albeit with very slight time differences. However, sounds and fx of the remote player is regulated by the local clock (nextallowedfiretime) and will give the impression of an exact refire rate. With moving targets, sometimes (though very rarely) this means a bullet misses on one client and hits on another. In this case it is always the master client's version of reality that is the law of the land. The master determines who lives or dies.

Misc problems and crashes

Strange things are happening / hard to find bugs

For example: One or more of the players are suddenly invisible to a certain client. Your game may have crashed and may be running in a "bad state". Run the game twice, once with the editor as master and one time with the editor as client. watch out for crashes, errors and warnings in the console. One infamous aspect of Unity is that when games crash, they will happily keep running (just with peculiar things starting to happen).

The WindowRenamer does not seem to work. Some windows don't get named correctly.

Make sure that "WindowRenamer -> Window name on startup" is exactly the same as the one set in "Unity ->  Edit -> Project Settings -> Player -> Product Name", AND that there are no other windows open on your desktop that have the same name. For example: if you have a Visual Studio project open with the name "MyGame", you must not use the string "MyGame" as the Unity Product Name.

Game seems to enforce singleplayer. Scoreboard does not show, noone can join

Is the master running as a standalone executable? It may have crashed and silently broken the simulation. Unity webplayers and exes have the infamous ability of crashing silently with the game running in a broken state but without any error messages. Being client to a master in that state won't be an enjoyable experience.

An RPC does not execute on the correct gameobject

  • photonView.RPC = executes a method on the specific gameobject of that specific photonview
  • The "PhotonTargets" parameter means "on whose machines should this specific photonview be affected"?
  • If the following is run on the masterclient:
  • photonView.RPC -> will affect the gameobject of the masterclient object
  • player.photonView.RPC -> will affect the gameobject of player

The standalone executable crashes with a unity dialog saying "Oops - the game crashed"

Unity crashes when starting the game

This can obviously happen for a number of reasons, but we've noticed an odd tendency of either UFPS, Photon Unity Networking or Unity 4.3 (don't know which one yet) to sometimes produce broken builds on the first session. That is, when the editor is launched for the first time since computer boot, Unity will crash when launching the game, and any executable built when the editor is in this state will also crash on launch. If the editor is restarted and the game built anew, everything will work fine.

Error Messages

"Error (vp_MPPlayerSpawner) Failed to get player type '<name>'. Make sure this object exists in your MPPlayerSpawner->PlayerTypes list."

If you have made sure the object exists in the list (as suggested by the error message) but the error persists, then verify that your vp_MPMaster or vp_DMMaster component is listed _below_ your vp_MPPlayerSpawner component in the Inspector. If not, there will be a problem with script initialization order upon level load. In Unity you can move components up and down on gameobjects by clicking the little cog wheel on the component -> "Move Up / Down"

  • Is this duplicate ID used on either the GUI (main menu) gameobject or the master gameobject (the one with the vp_MPMaster or vp_DMMaster component + vp_MPConnection component)? If so, make sure to set a high number for these gameobjects. For example: 900 and 901. For example, if the PhotonView ID on your main menu gameobject is identical to the one of, say, a crate in a level that gets loader later, then Photon will post an error. Players automatically get PhotonView IDs assigned starting at 1001 for player one, 2001 for player two etc. Regular scene gameobjects automatically get assigned IDs starting at 1, which is usually the problem. One thing you can do to avoid this is set a very high number (below 1000) for any standard gameobjects that are to be retained between levels. For example: 900-1000, minimizing the risk of colliding with things like in-level rigidbodies.

"Level '<levelname>' (-1) couldn't be loaded because it has not been added to the build settings."

The multiplayer starter kit demos rely on Unity's level loading feature. Before the game can load any level you must add it to the Unity Editor build settings. From the Unity menu go to File -> Build Settings and tick the appropriate level name checkboxes in the list (Tip: the demo scenes all have names beginning with "UFPSMP_"). Note that you don't have to click "Build". Just check the checkbox(es), close the window and try running the game again. The main menu scenes must always be the topmost one (index 0).

"Received RPC "<RPCName>" for viewID <id> but this PhotonView does not exist! Was remote PV. Owner called."

  • When sending RPCs, never use "PhotonTargets.OthersBuffered". This is not supported by the current UFPS multiplayer implementation. If you do, the game will crash for joining clients.
  • Make sure that no old / outdated client window is running in the background.
  • Verify that you are loading the same level on both clients. Sometimes this occurs when you are running one level in the editor but have forgotten to include that level in the build settings, so the standalone is running another level.
  • In the case of "FireWeapon" this is likely a harmless warning with little or no performance overhead. It will happen in the scene of a joining client if someone is firing while you log in. In this case, an RPC for a remotely fired weapon is sent to your scene before the objects involved have been set up, meaning there will be no recipient of the RPC.
  • If you find this message annoying, the Photon team recommends just locating the line in PUN that outputs this warning and commenting it out. At time of writing this warning is output from NetworkingPeer.cs -> ExecuteRPC().

"PhotonView with ID 2001 has no method "<RPCName>" marked with the [RPC](C#) or @RPC(JS) property! Args:"

Try making the RPC method public.

"ERROR You have missing MonoBehaviours on your gameobjects!"

This can occur when you have built a webplayer build and switched back to standalone. If scripts contain a build time define e.g.

#if UNITY_STANDALONE_WIN

... then they may need to be reassigned to their components after the build target switch in order to recompile. An example of a script that needs reassignment is vp_ClientWindowRenamer.

"Illegal view ID:0 method: Respawn GO:LocalPlayer(Clone)"

This occurs on startup if the respawner component of the local player is set to "SpawnOnAwake". Disable that checkbox and it should be fine.

"ArgumentException: An element with the same key already exists in the dictionary."

You may have tried to assemble / broadcast a game state where the same player ID was included twice or more. This won't work because the hashtable will crash: hastables can only contain unique keys
